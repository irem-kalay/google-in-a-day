# Production Roadmap

## What the Current System Does Well — and Where It Falls Short

This system works well as a prototype. It crawls pages concurrently, indexes content in real time, and handles back-pressure through a bounded queue. However, it has some weaknesses that would cause serious problems in a production environment. First, the entire visited set and inverted index live in memory. If the process crashes or the machine runs out of RAM, all the indexed data is lost — the `state.json` file helps, but it is not a reliable solution for large-scale use. Second, the crawler does not respect `robots.txt` or apply any per-domain rate limiting, which means it can easily get IP-banned by websites. Third, the search algorithm is not so complex — it ranks results by keyword frequency, whether searched keywords are together or title matching, which is not enough to return truly relevant results for complex queries. Finally, the system starts from only a handful of seed URLs, which severely limits how much of the web it can actually discover and index.

## What Should Be Done for Production

To make this system production-ready, the in-memory data structures need to be replaced with proper storage solutions. The inverted index should be moved to a dedicated search engine like Elasticsearch, which handles ranking, scaling, and fault tolerance much better. The URL queue should be replaced with a distributed message broker like Redis or Kafka, so multiple machines can work together without stepping on each other. The crawler also needs politeness logic — delays between requests to the same domain and `robots.txt` parsing — to avoid getting blocked. The seed URL list should be expanded to hundreds or thousands of diverse, high-quality starting points so the crawler can discover a much wider range of pages from the beginning. Finally, the `state.json` persistence should be replaced with a real database like PostgreSQL so that no data is lost when something goes wrong. These changes would turn this prototype into a system that can actually run at scale without constant manual intervention.
